# 设计模式（GoF 23种模式分类）

## 创建型模式（5种）

* **单例（Singleton）**
  * 懒汉式：延迟加载，需考虑线程安全
  * 饿汉式：类加载时即创建，天然线程安全
  * 双重检查锁定（DCL）：减少同步开销
  ```cpp
  class Singleton {
  private:
      static Singleton* instance;
      static std::mutex mtx;
  public:
      static Singleton* getInstance() {
          if (instance == nullptr) {
              std::lock_guard<std::mutex> lock(mtx);
              if (instance == nullptr)
                  instance = new Singleton();
          }
          return instance;
      }
  };
  ```

* **工厂方法（Factory Method）**
  * 定义一个创建对象的接口，让子类决定实例化哪个类
  * 避免在代码中硬编码具体类
  * UML: Creator → Factory Method() → Product

* **抽象工厂（Abstract Factory）**
  * 创建一组相关或相互依赖的对象
  * 产品族的概念：如Windows/Mac不同风格的按钮+文本框

* **建造者（Builder）**
  * 将一个复杂对象的构建与表示分离
  * 适用于构造参数多且可选的情况（如构建SQL查询、HTTP请求）

* **原型（Prototype）**
  * 通过拷贝现有对象来创建新对象（深拷贝和浅拷贝）
  * 适用于对象创建成本高的情况

## 结构型模式（7种）

* **适配器（Adapter）**
  * 使不兼容的接口能够一起工作
  * 类适配器（继承）vs 对象适配器（组合）
* **装饰器（Decorator）**
  * 动态给对象添加职责，比继承更灵活
  * 典型应用：Java IO 流（BufferedInputStream装饰FileInputStream）
* **代理（Proxy）**
  * 控制对真实对象的访问（延迟加载、访问控制、日志记录）
  * 远程代理、虚拟代理、保护代理、智能引用
* **外观（Facade）**
  * 为子系统提供统一的高层接口
* **桥接（Bridge）**
  * 将抽象部分与实现部分分离
* **组合（Composite）**
  * 树形结构：部分-整体层次结构
* **享元（Flyweight）**
  * 共享细粒度对象，减少内存占用

## 行为型模式（11种）

* **策略（Strategy）**
  * 定义一系列算法，使它们可以互相替换
  * 典型应用：排序算法的可插拔策略
* **观察者（Observer）**
  * 定义一对多依赖关系，状态变化时通知所有观察者
  * 应用：事件监听、消息队列、MVC架构
* **模板方法（Template Method）**
  * 在父类定义算法骨架，子类实现具体步骤
* **责任链（Chain of Responsibility）**
  * 多个处理对象连成一条链，依次处理请求
  * 应用：中间件（Express/Koa）、日志处理
* **命令（Command）**
  * 将请求封装为对象，支持撤销、重做、队列
* **迭代器（Iterator）**
  * 提供顺序访问聚合对象元素的方法
* **状态（State）**
  * 对象内部状态改变时改变其行为，如TCP连接状态机
* **访问者（Visitor）**
  * 在不改变元素类的前提下，定义对元素的新操作
* **中介者（Mediator）**
  * 减少多个对象之间的直接通信
* **备忘录（Memento）**
  * 保存和恢复对象的内部状态（撤销操作）
* **解释器（Interpreter）**
  * 定义语言的文法，并解释语言中的句子

## 设计原则 SOLID

* **S** - 单一职责原则（Single Responsibility）
  * 一个类只负责一个功能领域的职责
* **O** - 开闭原则（Open/Closed）
  * 对扩展开放，对修改关闭
* **L** - 里氏替换原则（Liskov Substitution）
  * 子类可以替换父类，但不改变程序的正确性
* **I** - 接口隔离原则（Interface Segregation）
  * 接口应该小而专，避免胖接口
* **D** - 依赖倒转原则（Dependency Inversion）
  * 依赖抽象接口而非具体实现

## 架构模式详解

### 分层架构

* 表现层（UI）→ 应用层（Service）→ 领域层（Domain）→ 基础设施层（Infrastructure）
* 上层依赖下层，下层不依赖上层（依赖倒置可解耦）
* **优点**：结构清晰、关注点分离
* **缺点**：大项目层数过多导致修改成本高

### 事件驱动架构（EDA）

* 组件通过事件异步通信，不直接调用
* **核心组件**：事件生产者、事件总线/消息队列、事件消费者
* **事件溯源（Event Sourcing）**：存储所有状态变更事件而非当前状态
* **CQRS（命令查询职责分离）**：写操作和读操作使用不同的模型

### 六边形架构（Ports & Adapters）

* 核心业务逻辑不依赖任何外部系统（数据库、Web、消息队列）
* 通过 Port（接口）和 Adapter（实现）连接外部
* 测试友好：可以 mock Adapter 测试核心逻辑

### 整洁架构（Clean Architecture）

* 同心圆模型：最内层是实体（Entities），往外是用例（Use Cases），再往上是接口适配器，最外层是框架和驱动
* **依赖规则**：依赖只能从外向内，内部不能知道外部

## 领域驱动设计（DDD）

### 战略设计

* **限界上下文（Bounded Context）**：每个领域概念的边界范围
* **通用语言（Ubiquitous Language）**：团队统一使用的领域术语
* **上下文映射（Context Map）**：不同限界上下文之间的关系

### 战术设计

| 概念 | 说明 |
|------|------|
| 实体（Entity） | 有唯一标识，可改变状态（如用户、订单） |
| 值对象（Value Object） | 无标识，不可变（如金额、地址） |
| 聚合（Aggregate） | 实体+值对象的组合，有根（Aggregate Root） |
| 领域事件（Domain Event） | 领域中发生的重要事件 |
| 仓储（Repository） | 聚合的持久化接口 |
| 工厂（Factory） | 复杂对象的创建 |
| 服务（Domain Service） | 不属于任何实体/值对象的业务逻辑 |

## 代码重构

### 何时重构

* 代码有坏味道：过长函数、过大类、重复代码、switch 语句、基本类型偏执
* 添加新功能前（先重构再实现）
* 修改 Bug 时（理解代码后顺便清理）
* Code Review 发现可改进点

### 常用重构手法

| 手法 | 说明 |
|------|------|
| Extract Method | 将代码段提取为独立函数 |
| Rename Variable/Function | 命名是代码可读性的关键 |
| Replace Magic Number | 用常量替代魔法数字 |
| Replace Conditional with Polymorphism | 用多态替代 if-else/switch |
| Introduce Parameter Object | 多个参数封装为对象 |
| Pull Up / Push Down | 继承体系中向上/下移动成员 |
| Decompose Conditional | 分解复杂的条件表达式 |

### 重构安全

* 小步提交：每次只做一种重构，随时可回退
* 测试覆盖：重构前确保有足够的测试
* 重构 ≠ 重写：不改变外部行为
* 使用 IDE 重构工具：安全可靠

## 技术债务

* **定义**：为短期利益做出的妥协性设计/实现，未来需要偿还
* **分类**：
  * 有意的：明知需要改进但先交付
  * 无意的：缺乏经验或知识不足
  * 过期的：早期决策不再适用
* **管理**：像财务债务一样跟踪，定期偿还利息（小重构）
* **度量**：圈复杂度、重复代码率、测试覆盖率、静态分析问题数

## 分布式系统理论

### CAP 定理

* **C（一致性）**：所有节点在同一时刻看到相同的数据
* **A（可用性）**：每次请求都能得到非错误的响应
* **P（分区容错性）**：系统在部分节点网络断开时仍能工作

在实际分布式系统中，**P 是必须的**，需要在 C 和 A 之间权衡：

| 选择 | 解释 | 典型系统 |
|------|------|---------|
| CP | 分区时放弃可用性，保证一致性 | Zookeeper、etcd |
| AP | 分区时放弃一致性，保证可用性 | Eureka、Cassandra |
| CA | 不允许分区（实际不存在） | 单机数据库 |

### BASE 理论

* **BA（Basically Available）**：基本可用（允许降级）
* **S（Soft State）**：状态可以在一段时间内不一致
* **E（Eventually Consistent）**：最终一致性

### 一致性模型

| 模型 | 说明 | 影响 |
|------|------|------|
| 强一致性 | 写完后所有读都能看到 | 性能最差 |
| 最终一致性 | 一段时间后所有副本一致 | DNS、CDN 缓存 |
| 因果一致性 | 有因果关系的操作顺序一致 | 评论回复场景 |
| 单调读一致性 | 同一客户端不会读到更旧的数据 | 大多数系统默认 |
| Read-Your-Writes | 客户端总能读到自己的写入 | 用户设置即时生效 |

### 分布式事务

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **2PC（两阶段提交）** | 准备阶段+提交阶段 | 实现简单 | 同步阻塞、协调者单点 |
| **3PC** | 引入超时机制减少阻塞 | 相对 2PC 更优 | 仍可能不一致 |
| **TCC（Try-Confirm-Cancel）** | 业务层面补偿，Try预留资源 | 高性能、无锁 | 业务侵入性强 |
| **Saga** | 每个操作有补偿操作，失败时反向恢复 | 适合长事务 | 事务隔离性弱 |

## 项目管理实践

### 敏捷估算方法

| 方法 | 说明 | 单位 |
|------|------|------|
| 故事点（Story Point） | 综合复杂度、工作量、不确定性 | 1/2/3/5/8/13（斐波那契） |
| 人天（Man-Day） | 按开发人员的工作日估算 | 人天/人时 |
| 理想工时 | 完全不被打扰的工作时间 | 小时 |

* **扑克牌估算（Planning Poker）**：团队每人出一张牌，差距大时讨论，得出共识
* **速度（Velocity）**：团队在一个 Sprint 中实际完成的故事点

### 站会/回顾会

* **每日站会（15分钟）**：
  * 昨天完成了什么？
  * 今天计划做什么？
  * 遇到什么阻碍？
* **Sprint 回顾会**：
  * 做得好的（Keep）
  * 需要改进的（Problem）
  * 下一步行动计划（Try）

### OKR vs KPI

| 维度 | OKR | KPI |
|------|-----|-----|
| 目的 | 挑战性目标，方向对齐 | 可量化指标，考核评估 |
| 设定 | 自上而下+自下而上结合 | 自上而下分解 |
| 挑战性 | 完成 70% 就算成功 | 要求 100% 完成 |
| 关联 | 不直接与薪酬挂钩 | 通常与绩效挂钩 |
