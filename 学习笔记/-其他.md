# 设计模式（GoF 23种模式分类）

## 创建型模式（5种）

* **单例（Singleton）**
  * 懒汉式：延迟加载，需考虑线程安全
  * 饿汉式：类加载时即创建，天然线程安全
  * 双重检查锁定（DCL）：减少同步开销
  ```cpp
  class Singleton {
  private:
      static Singleton* instance;
      static std::mutex mtx;
  public:
      static Singleton* getInstance() {
          if (instance == nullptr) {
              std::lock_guard<std::mutex> lock(mtx);
              if (instance == nullptr)
                  instance = new Singleton();
          }
          return instance;
      }
  };
  ```

* **工厂方法（Factory Method）**
  * 定义一个创建对象的接口，让子类决定实例化哪个类
  * 避免在代码中硬编码具体类
  * UML: Creator → Factory Method() → Product

* **抽象工厂（Abstract Factory）**
  * 创建一组相关或相互依赖的对象
  * 产品族的概念：如Windows/Mac不同风格的按钮+文本框

* **建造者（Builder）**
  * 将一个复杂对象的构建与表示分离
  * 适用于构造参数多且可选的情况（如构建SQL查询、HTTP请求）

* **原型（Prototype）**
  * 通过拷贝现有对象来创建新对象（深拷贝和浅拷贝）
  * 适用于对象创建成本高的情况

## 结构型模式（7种）

* **适配器（Adapter）**
  * 使不兼容的接口能够一起工作
  * 类适配器（继承）vs 对象适配器（组合）
* **装饰器（Decorator）**
  * 动态给对象添加职责，比继承更灵活
  * 典型应用：Java IO 流（BufferedInputStream装饰FileInputStream）
* **代理（Proxy）**
  * 控制对真实对象的访问（延迟加载、访问控制、日志记录）
  * 远程代理、虚拟代理、保护代理、智能引用
* **外观（Facade）**
  * 为子系统提供统一的高层接口
* **桥接（Bridge）**
  * 将抽象部分与实现部分分离
* **组合（Composite）**
  * 树形结构：部分-整体层次结构
* **享元（Flyweight）**
  * 共享细粒度对象，减少内存占用

## 行为型模式（11种）

* **策略（Strategy）**
  * 定义一系列算法，使它们可以互相替换
  * 典型应用：排序算法的可插拔策略
* **观察者（Observer）**
  * 定义一对多依赖关系，状态变化时通知所有观察者
  * 应用：事件监听、消息队列、MVC架构
* **模板方法（Template Method）**
  * 在父类定义算法骨架，子类实现具体步骤
* **责任链（Chain of Responsibility）**
  * 多个处理对象连成一条链，依次处理请求
  * 应用：中间件（Express/Koa）、日志处理
* **命令（Command）**
  * 将请求封装为对象，支持撤销、重做、队列
* **迭代器（Iterator）**
  * 提供顺序访问聚合对象元素的方法
* **状态（State）**
  * 对象内部状态改变时改变其行为，如TCP连接状态机
* **访问者（Visitor）**
  * 在不改变元素类的前提下，定义对元素的新操作
* **中介者（Mediator）**
  * 减少多个对象之间的直接通信
* **备忘录（Memento）**
  * 保存和恢复对象的内部状态（撤销操作）
* **解释器（Interpreter）**
  * 定义语言的文法，并解释语言中的句子

## 设计原则 SOLID

* **S** - 单一职责原则（Single Responsibility）
  * 一个类只负责一个功能领域的职责
* **O** - 开闭原则（Open/Closed）
  * 对扩展开放，对修改关闭
* **L** - 里氏替换原则（Liskov Substitution）
  * 子类可以替换父类，但不改变程序的正确性
* **I** - 接口隔离原则（Interface Segregation）
  * 接口应该小而专，避免胖接口
* **D** - 依赖倒转原则（Dependency Inversion）
  * 依赖抽象接口而非具体实现

## 架构模式详解

### 分层架构

* 表现层（UI）→ 应用层（Service）→ 领域层（Domain）→ 基础设施层（Infrastructure）
* 上层依赖下层，下层不依赖上层（依赖倒置可解耦）
* **优点**：结构清晰、关注点分离
* **缺点**：大项目层数过多导致修改成本高

### 事件驱动架构（EDA）

* 组件通过事件异步通信，不直接调用
* **核心组件**：事件生产者、事件总线/消息队列、事件消费者
* **事件溯源（Event Sourcing）**：存储所有状态变更事件而非当前状态
* **CQRS（命令查询职责分离）**：写操作和读操作使用不同的模型

### 六边形架构（Ports & Adapters）

* 核心业务逻辑不依赖任何外部系统（数据库、Web、消息队列）
* 通过 Port（接口）和 Adapter（实现）连接外部
* 测试友好：可以 mock Adapter 测试核心逻辑

### 整洁架构（Clean Architecture）

* 同心圆模型：最内层是实体（Entities），往外是用例（Use Cases），再往上是接口适配器，最外层是框架和驱动
* **依赖规则**：依赖只能从外向内，内部不能知道外部

## 领域驱动设计（DDD）

### 战略设计

* **限界上下文（Bounded Context）**：每个领域概念的边界范围
* **通用语言（Ubiquitous Language）**：团队统一使用的领域术语
* **上下文映射（Context Map）**：不同限界上下文之间的关系

### 战术设计

| 概念 | 说明 |
|------|------|
| 实体（Entity） | 有唯一标识，可改变状态（如用户、订单） |
| 值对象（Value Object） | 无标识，不可变（如金额、地址） |
| 聚合（Aggregate） | 实体+值对象的组合，有根（Aggregate Root） |
| 领域事件（Domain Event） | 领域中发生的重要事件 |
| 仓储（Repository） | 聚合的持久化接口 |
| 工厂（Factory） | 复杂对象的创建 |
| 服务（Domain Service） | 不属于任何实体/值对象的业务逻辑 |

## 代码重构

### 何时重构

* 代码有坏味道：过长函数、过大类、重复代码、switch 语句、基本类型偏执
* 添加新功能前（先重构再实现）
* 修改 Bug 时（理解代码后顺便清理）
* Code Review 发现可改进点

### 常用重构手法

| 手法 | 说明 |
|------|------|
| Extract Method | 将代码段提取为独立函数 |
| Rename Variable/Function | 命名是代码可读性的关键 |
| Replace Magic Number | 用常量替代魔法数字 |
| Replace Conditional with Polymorphism | 用多态替代 if-else/switch |
| Introduce Parameter Object | 多个参数封装为对象 |
| Pull Up / Push Down | 继承体系中向上/下移动成员 |
| Decompose Conditional | 分解复杂的条件表达式 |

### 重构安全

* 小步提交：每次只做一种重构，随时可回退
* 测试覆盖：重构前确保有足够的测试
* 重构 ≠ 重写：不改变外部行为
* 使用 IDE 重构工具：安全可靠

## 技术债务

* **定义**：为短期利益做出的妥协性设计/实现，未来需要偿还
* **分类**：
  * 有意的：明知需要改进但先交付
  * 无意的：缺乏经验或知识不足
  * 过期的：早期决策不再适用
* **管理**：像财务债务一样跟踪，定期偿还利息（小重构）
* **度量**：圈复杂度、重复代码率、测试覆盖率、静态分析问题数
