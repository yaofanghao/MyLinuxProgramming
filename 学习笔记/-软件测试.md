### 安全测试扩展

* **OWASP Top 10 (2021)**
  * A01: 权限控制失效（Broken Access Control）
  * A02: 加密机制失效（Cryptographic Failures）
  * A03: 注入攻击（Injection）
  * A04: 不安全设计（Insecure Design）
  * A05: 安全配置错误（Security Misconfiguration）

* **SQL 注入防御**
  * 使用"参数化查询"（PreparedStatement），禁止拼接SQL字符串
  * 输入验证：白名单策略
  * 最小权限原则：数据库账户仅赋予必要权限

* **XSS 跨站脚本攻击**
  * 反射型、存储型、DOM型
  * 防御：输出编码（HTML Entity）、CSP（内容安全策略）、HttpOnly Cookie

* **CSRF 跨站请求伪造**
  * 防御：CSRF Token、SameSite Cookie 属性、Referer 校验

### 自动化测试架构

* **测试金字塔（Mike Cohn）**
  * 底层：大量单元测试（快速、稳定）
  * 中间层：少量服务/API测试
  * 顶层：极少端到端UI测试（慢、脆弱）
  * 建议比例：70% 单元 : 20% 服务 : 10% E2E

* **Page Object Model（POM）**
  * 将页面元素和操作封装为 Page 类
  * 测试用例只调用业务方法，不直接操作元素
  * 页面变化时只需修改 Page 类，不影响测试用例

* **数据驱动测试**
  * 测试逻辑与测试数据分离
  * 数据源：CSV、Excel、YAML、JSON、数据库

### 性能测试扩展

* **性能分析思路**
  * 自上而下：CPU → 内存 → 磁盘I/O → 网络
  * 自下而上：代码热点 → 数据库查询 → 外部调用
  * 工具链：top/htop → perf → strace → flamegraph

* **常见性能瓶颈**
  * 数据库：缺少索引、慢查询、连接池不够
  * 代码：序列化/反序列化开销、锁竞争、内存分配频繁
  * 系统：句柄泄漏、内存泄漏、CPU 上下文切换过高

### 接口测试

* **核心验证点**
  * 状态码：2xx/4xx/5xx 是否符合预期
  * 响应体：数据结构、字段类型、取值范围
  * 响应时间：P99/P95 延迟
  * 幂等性：多次调用结果一致
  * 边界：空参数、超长参数、特殊字符、枚举越界

* **工具对比**
  | 工具 | 特点 | 适用场景 |
  |------|------|---------|
  | Postman | GUI 友好，支持集合/环境变量 | 手动测试、调试 |
  | Newman | Postman 集合命令行运行 | CI/CD 集成 |
  | curl | 最基础，无依赖 | 快速验证 |
  | JMeter | 支持参数化、断言、报告 | 接口性能测试 |
  | pytest + requests | Python 脚本，灵活 | 自动化接口测试 |

### 压力测试

* **QPS（Queries Per Second）**：每秒请求数
* **TPS（Transactions Per Second）**：每秒事务数
* **并发用户数**：同时操作系统的用户数量
* **吞吐量**：单位时间内系统处理的请求数
* **思考时间**：用户两次操作之间的间隔

* **常见压测工具**
  * **wrk**：轻量级 HTTP 压测，支持 Lua 脚本
  * **ab（Apache Bench）**：简单压测
  * **JMeter**：功能全面，支持分布式压测
  * **Locust**：Python 编写，可编程
  * **sysbench**：数据库压测

### 代码覆盖率

| 覆盖类型 | 说明 | 要求 |
|---------|------|------|
| 行覆盖（Line） | 每行代码是否执行过 | 基础要求 |
| 分支覆盖（Branch） | 每个 if/else 分支是否执行 | 较高要求 |
| 条件覆盖（Condition） | 每个布尔子表达式是否取过 true/false | 更严格 |
| 路径覆盖（Path） | 所有可能的执行路径 | 理论最优，实际难做到 |
| 函数覆盖（Function） | 每个函数是否调用过 | 基本要求 |

* **工具**：gcov（C/C++）、JaCoCo（Java）、coverage.py（Python）

### 测试流程规范

* **测试计划包含**
  * 测试范围与目标
  * 资源与时间安排
  * 测试环境与数据准备
  * 准入准出标准
  * 风险与应急预案

* **Bug 生命周期**
  ```
  新建(New) → 确认(Confirmed) → 分配给开发者(Assigned)
  → 修复(Resolved/Fixed) → 验证(Verified/Closed)
  → 重新打开(Reopened)
  ```

* **测试报告要素**
  * 测试执行概况（总用例/通过/失败/阻塞数）
  * Bug 统计（按严重级别/模块分布）
  * 风险遗留问题
  * 版本质量评估结论
  * 改进建议
