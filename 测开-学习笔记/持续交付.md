## 敏捷开发 vs 传统瀑布模型

| 维度 | 瀑布模型 | 敏捷开发 |
|------|---------|---------|
| 需求 | 前期完整确定 | 迭代演进 |
| 开发周期 | 数月到数年 | 1-4周冲刺 |
| 交付 | 一次性交付 | 持续交付小版本 |
| 变更响应 | 困难、成本高 | 灵活、成本低 |
| 客户参与 | 需求阶段后少参与 | 全程紧密参与 |
| 测试 | 开发完成后集中测试 | 持续集成、持续测试 |
| 文档 | 大量文档 | 适度文档、可工作软件优先 |

## 微服务架构设计要点

### 服务拆分原则

* **领域驱动设计（DDD）**：按业务领域拆分，每个服务对应一个限界上下文
* **单一职责**：每个服务只负责一个业务能力
* **无共享架构**：服务间通过 API 通信，不共享数据库
* **数据独立**：每个服务拥有自己的数据存储

### 微服务通信方式

| 方式 | 协议 | 特点 | 适用场景 |
|------|------|------|---------|
| RESTful API | HTTP/JSON | 简单、易调试 | 同步请求-响应 |
| gRPC | HTTP/2 + Protobuf | 高性能、强类型 | 内部服务间高频调用 |
| 消息队列 | AMQP/Kafka | 异步、削峰填谷 | 事件驱动、解耦 |
| GraphQL | HTTP | 客户端灵活查询 | 复杂数据聚合 |

### 微服务治理

* **服务发现**：Consul、Eureka、Nacos
* **负载均衡**：Ribbon、Nginx、Envoy
* **熔断降级**：Hystrix、Resilience4j、Sentinel
* **API 网关**：Kong、Spring Cloud Gateway、APISIX
* **链路追踪**：Jaeger、Zipkin、SkyWalking

## 容器化与 K8s 部署流水线

### 镜像构建流水线示例（GitHub Actions）

```yaml
name: Build and Deploy
on:
  push:
    branches: [main]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Login to Docker Hub
        run: docker login -u ${{ secrets.DOCKER_USER }} -p ${{ secrets.DOCKER_PASS }}
      - name: Build and push
        run: |
          docker build -t myapp:${{ github.sha }} .
          docker push myapp:${{ github.sha }}
      - name: Deploy to K8s
        run: |
          kubectl set image deployment/myapp myapp=myapp:${{ github.sha }}
```

### 蓝绿部署 vs 滚动更新 vs 金丝雀发布

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **蓝绿部署** | 两套完整环境，切换流量 | 秒级回滚、环境隔离 | 资源消耗翻倍 |
| **滚动更新** | 逐步替换旧版本实例 | 资源消耗不变、平滑过渡 | 回滚慢、兼容性要求高 |
| **金丝雀发布** | 小比例流量跑新版本 | 风险最小、真实验证 | 需要流量治理能力 |

## 混沌工程

* 在生产环境引入故障来验证系统的韧性
* 核心原则
  * **稳态假设**：定义系统的"正常"行为指标
  * **控制变量**：控制爆炸半径，避免影响真实用户
  * **持续运行**：自动化、持续化实验

* 常见实验
  * 杀死 Pod（K8s Chaos Mesh）
  * 网络延迟/丢包（tc/netem）
  * CPU/内存压力（stress-ng）
  * 依赖服务超时/返回错误

* 工具：Chaos Mesh、Litmus、Gremlin

## 可观测性（Observability）

### 三大支柱

* **指标（Metrics）**
  * Prometheus + Grafana
  * RED 指标：Rate（速率）、Errors（错误）、Duration（耗时）
  * USE 指标：Utilization（利用率）、Saturation（饱和度）、Errors（错误）

* **日志（Logging）**
  * ELK Stack：Filebeat → Logstash → Elasticsearch → Kibana
  * Loki：轻量级日志聚合，与 Prometheus 集成

* **链路追踪（Tracing）**
  * OpenTelemetry：统一的数据采集标准
  * 采样策略：头部采样、尾部采样、概率采样

### SLI / SLO / SLA

* **SLI（服务级别指标）**：可用性、延迟、吞吐量、错误率
* **SLO（服务级别目标）**：99.9% 请求延迟 < 200ms
* **SLA（服务级别协议）**：对外承诺，通常附带赔偿条款
